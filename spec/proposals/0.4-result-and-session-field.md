# Proposal: Wire 0.4 — `result` event, `session_id` as event field

**Status:** Merged into `spec/protocol.md` / `spec/protocol.zh.md` (wire 0.4 / doc 1.1). This file is retained as the design record; the protocol documents are normative.

**Supersedes:** wire 0.3 event vocabulary for stdout session / final body
**Date:** 2026-07-14

---

## Motivation

1. **`{"type":"session"}` is the wrong abstraction.** Session id is turn-level metadata, not a discrete “thing that happened.” Hub bridges that wrap NDJSON CLIs already receive `session_id` / `sessionID` / `thread_id` *on* CLI events and then reify a separate AgentProc session event — an extra translation step with no information gain.

2. **“Last wins” is hard to explain** because it names a conflict-resolution rule for a dedicated event type. The underlying need is only: “remember the session id once the agent has one.” Putting the id on events makes that rule either unnecessary (same id on every event) or trivial (last non-empty, for edge cases).

3. **`text` underspecifies a terminal turn outcome.** Coding CLIs terminate with a `result`-shaped event that carries body + session + usage together. Renaming the final body event to `result` matches that shape and gives a natural home for optional stats.

4. **Empirical check of hub CLIs (2026-07-14):** For resume-capable CLIs, the id is available early (`system/init`, `init`, `thread.started`, or on every event). The 0.3 Design Rationale claim that “the CLI often doesn’t know its session id until it exits” overstates the common case; e.g. `hub/claude-code` currently reads `session_id` only from `result`, while Claude’s `system/init` typically already carries it. Stateless profiles (`aider`, `deepseek`, `pi`, `agy`, `echo-agent`) correctly emit no session at all today — that honesty should be preserved (omit field; do **not** mint a fake id).

---

## Summary of changes (0.3 → 0.4)

| 0.3 | 0.4 |
|-----|-----|
| `{"type":"session","id":...}` | **Removed.** Use `session_id` field on events. |
| `{"type":"text","text":...}` (may repeat; concatenate) | **`{"type":"result","text":...}`** — at most one per turn. |
| Session rule: last `session` event wins | If any event carries `session_id`, every event in the turn MUST carry the same non-empty value. Stateless agents omit the field entirely. |
| No structured usage | Optional `usage` object on `result` / `error`. |
| `protocol_version`: `"0.3"` | `"0.4"` |

Unchanged in spirit: `partial`, `error`, `permission_request` / `permission_response`, one-turn-per-process, closed vocabulary otherwise.

## Output event vocabulary (0.4)

Closed set on stdout: `partial` | `result` | `error` | `permission_request`.

### Common optional / conditional fields

| Field | Type | Rule |
|-------|------|------|
| `session_id` | string | **Either omitted on every event of the turn, or present and non-empty on every event of the turn.** Empty string is invalid on output. When present, all events in the turn MUST use the same value. The bridge persists that value and passes it back as `turn.session_id` on the next turn. |
| `usage` | object | Optional. Meaningful on terminal events (`result`, `error`). Bridges MAY ignore. Forward-compatible: unknown keys inside `usage` SHOULD be ignored. |

**Stateless agents** (no native resume): omit `session_id` on all events. Do **not** mint a UUID solely to satisfy the field. Input `turn.session_id: ""` continues to mean “new / no prior session” on the way in; that is distinct from “agent does not support sessions” on the way out (signalled by omission).

**Resume-capable agents:** once the underlying CLI has produced an id, every subsequent AgentProc event MUST include it. Practical implication: buffer stdout translation until the id is known (typically the first CLI `init` / `thread.started` / equivalent), then flush `partial`s with `session_id` set. Do not emit AgentProc events with a missing or empty `session_id` in a turn that will later declare one.

### `{"type":"partial"}` — streaming chunk

```json
{"type":"partial","text":"hello ","role":"output","session_id":"cli-sess-…"}
```

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | Literal `"partial"`. |
| `text` | string | Chunk text. |
| `role` | string | Optional. `"output"` (default) or `"thinking"`. |
| `session_id` | string | Per common rule above. |

### `{"type":"result"}` — terminal success body

```json
{
  "type": "result",
  "text": "Here is the complete answer.",
  "session_id": "cli-sess-…",
  "usage": {"input_tokens": 12, "output_tokens": 34}
}
```

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | Literal `"result"`. |
| `text` | string | Final reply body for this turn. MAY be `""` when the full body was already delivered via `partial` (bridge skips a duplicate final send if appropriate). |
| `session_id` | string | Per common rule. |
| `usage` | object | Optional stats (see below). |

**At most one `result` per turn.** Bridges MUST ignore subsequent `result` events (or treat as protocol violation + warn) — TBD at implementation time; draft preference: warn + ignore after the first.

A turn that streams only `partial`s and exits `0` without a `result` remains a **successful** turn (same as 0.3 with no `text`). Resume-capable agents that omit `result` MUST still have placed `session_id` on every `partial` (and on `error` if any). Agents that need to publish `usage` without extra body SHOULD emit `{"type":"result","text":"","session_id":"…","usage":{…}}`.

### `{"type":"error"}` — terminal failure

```json
{
  "type": "error",
  "message": "Upstream API rate limited.",
  "session_id": "cli-sess-…",
  "usage": {"input_tokens": 12, "output_tokens": 0}
}
```

Same terminality rules as 0.3 (stop forwarding further `partial`s; discard later body; turn failed even if exit 0). `session_id` / `usage` MAY appear here so a failed turn can still hand back continuity and stats (common CLI `result{is_error:true}` shape).

### `usage` shape (non-normative keys; reserved names)

Recommended keys (all optional numbers or null):

```json
{
  "input_tokens": 0,
  "output_tokens": 0,
  "total_tokens": 0
}
```

Agents MAY add further keys (e.g. `cache_read_tokens`, `cost_usd`). Bridges that do not understand a key ignore it. The protocol does **not** add a separate `usage` event type.

## Bridge persistence rule (replaces “last wins”)

1. Persist the **first** non-empty `session_id` observed in the turn for the next turn’s `turn.session_id`.
2. If the turn never carries a non-empty `session_id` → do not learn a new agent session id from this turn.
3. Early events MAY omit `session_id`; once known, agents SHOULD attach it to subsequent events. Omitting-then-presenting is fine.
4. A **different** non-empty value later → protocol violation; warn; keep the first. Do not invent an id.

There is no “last wins” narrative: the first committed id wins; later conflict is a bug.

## Migration notes for hub / SDK authors

- Replace `emit_session(id)` + later `emit_text(body)` with `emit_partial(..., session_id=id)` / single `emit_result(text, session_id=id, usage=...)`.
- `claude-code` (and similar): parse `session_id` from `system/init` (or earliest event), then attach to every AgentProc event; do not wait for CLI `result` solely to learn the id.
- Stateless profiles: unchanged observable behaviour except `text` → `result` rename when they emit a final body.
- SDK `AgentResult.session_id` / return `{ sessionId }` maps to fields on emitted events, not to a `session` event.
- Conformance scenarios that assert `type=session` or multi-`text` concatenation need rewriting.
- Wire string bump: stdin `protocol_version` becomes `"0.4"`. Per Versioning rules it remains an opaque string (no `>=` checks).

## Design rationale (to land in `protocol.md` when merged)

**Why not a dedicated `session` event?**  
Because session id is metadata on the turn’s stream, not an event in the chat sense. NDJSON objects already allow fields; the 0.2-era `AGENT_SESSION:` line required a separate line type, and 0.3 preserved that shape out of inertia.

**Why SHOULD attach once known (not MUST on every event)?**  
Adversarial review found all-or-nothing MUST deadlocks mid-turn `permission_request` if wrappers buffer for discovery, and conflicts with fail-soft “first non-empty.” Wire 0.4 uses first-non-empty persistence + SHOULD on subsequent events; early omit is allowed.

**Why omit (not mint, not `""`) when there is no session?**  
Minting a UUID implies continuity the underlying tool cannot honour. Output `""` would collide semantically with input `turn.session_id: ""` (new session). Omission is the honest capability signal. Generating an id a CLI requires as `--session` input is allowed.

**Why `result` instead of `text`?**  
Matches CLI terminal events; gives a single place for final body + `session_id` + `usage`; avoids “which of several `text` lines carries usage?”

**Why optional `usage` on `result`/`error` rather than a new event type?**  
Keeps the closed vocabulary small; stats ride with the terminal outcome the same way they do in upstream CLIs.
